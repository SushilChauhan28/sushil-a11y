Every design system ships components. Buttons, modals, tabs, chips, toolbars — assembled into products used by millions. And yet in almost every audit, the same failures appear across different teams, companies, and tech stacks.

The failures are rarely exotic. Missing aria-expanded. Focus not returning after a modal closes. A contrast ratio of 2.6:1 on an interactive icon. Tab doing the job that arrow keys should do inside a toolbar. These are not edge cases — they are the default state of unreviewed components.

This guide follows the same structure for every component: Common Issues (what was actually found in the audit), Best Practices (what the correct implementation requires), and Example Markup (ready-to-use ARIA code). Where something is particularly critical, a note follows.

Breadcrumbs
Navigation / Wayfinding
Common Issues

No clear indication that this is a breadcrumb trailshare The markup doesn’t tell screen readers or assistive tools that this is a breadcrumb navigation. It’s just a list, without any context.

The “Home” item is not interactive or labeled It’s sitting there as an empty list item with a class name, but it doesn’t do anything. It’s not clickable, not labeled, and not accessible to keyboard users.

No keyboard support cues There’s no sign that any part of this breadcrumb can be reached or used with a keyboard. If someone is navigating without a mouse, they won’t know where they are.

No label to describe the purpose of this list Screen readers won’t know this is a breadcrumb trail unless it’s explicitly labeled. Right now, it’s just a list of items with no context.

Empty list items with no meaning Some list items are present but contain no meaningful content. They don’t help users understand or navigate and may confuse screen readers.

Low contrast between separator and background The separator (arrow) uses a foreground color of #00B2A9 on a white background (#FFFFFF), resulting in a contrast ratio of 2.644:1, which is below the minimum required for normal text and non-decorative elements.

Best Practices

1. Use the nav Landmark with Descriptive Label

Wrap breadcrumbs inside a <nav> element.

Add aria-label="Breadcrumb" to describe the type of navigation.

This helps screen reader users understand the purpose of this navigation region.

2. Use an Ordered List (<ol>) for Breadcrumb Items

Breadcrumbs represent a hierarchical path, so use <ol>, not <ul>.

This communicates structure and reading order correctly.

3. Make the Home Icon Interactive and Accessible

The home icon should be a focusable link (<a>).

Provide an accessible name using either visible text or aria-label, e.g., aria-label="Home".

Ensure it has role="link" if necessary (usually implied by <a>).

Add a visible keyboard focus indicator (outline, underline) to help keyboard users identify focus.

Accordion
Disclosure / Expand–Collapse
Common Issues

Accordion is not wrapped inside heading semantics The accordion header (mat-expansion-panel-header) is not placed inside a heading element. This breaks the semantic structure of the page and makes it harder for screen readers to understand the hierarchy of content.

No visible focus indicator on the accordion header The accordion header is keyboard-focusable (tabindex="0"), but there is no visible focus style when it receives keyboard focus. This makes it difficult for keyboard users to know where they are on the page.

Missing heading structure for accordion title The title “TIMEKEEPER DETAILS” is not inside a heading tag (<h2>, <h3>, etc.). Accordions should follow the page’s heading structure to help screen reader users navigate efficiently.

Best Practices

Wrap Accordion Titles in Heading Elements

Use semantic headings (<h2>, <h3>, etc.) to structure accordion titles.

This helps screen reader users understand the hierarchy and navigate easily.

Use a Button Element Inside the Heading

The interactive part of the accordion should be a <button> inside the heading. If the heading text is placed inside a <button> instead of the button being placed inside a heading element (e.g., <h3><button>...</button></h3>), screen readers like JAWS may not announce it as a heading. This suppresses the semantic value of the heading and affects navigation for users relying on heading shortcuts.

Include aria-expanded = “true/false” and aria-controls to indicate state and relationship.

Make sure visually hidden content is not exposed to Assistive technology user.

Ensure Visible Focus Indicator

Provide a clear visual style (e.g., outline) when the accordion header is focused.

This helps keyboard users track their navigation.

Improve Color Contrast of Expand/Collapse Indicators

Ensure the arrow icon meets the minimum contrast ratio of 3:1.

Toggle Button
Mutually Exclusive vs. Independent States
Common Issues

Incorrect use of toggle button for mutually exclusive options The current implementation uses toggle buttons to represent three mutually exclusive options — Bold, Italic, and Underline — where only one can be selected at a time. This behavior is similar to radio buttons, not toggle buttons. Toggle buttons are typically used for independent on/off states, not grouped selections.

Pressed state is misused for radio-like behavior In a true toggle button, the user should be able to press and unpress the same button. But in this case, one option is always pressed, and pressing another unpresses the previous — mimicking radio button behavior. This can confuse users expecting toggle-like behavior.

Incorrect ARIA roles and states for grouped toggle buttons The group uses role="group" and individual buttons use aria-pressed, which is appropriate for standalone toggle buttons. However, for mutually exclusive selections, role="radiogroup" and role="radio" with aria-checked would be more semantically correct.

Best Practices

Use Radio Buttons for Mutually Exclusive Selections

If only one option can be selected at a time, use role="radiogroup" and role="radio" with aria-checked.

This ensures screen readers announce the correct behavior and state.

Use Toggle Buttons for Independent On/Off/Mixed States

Toggle buttons are best for standalone actions like “Play Sound” where the user can toggle the state on or off.

Use aria-pressed for these cases.

Ensure Visible Focus Indicators

Provide a clear visual style (e.g., outline or background change) when a toggle button receives focus.

This helps keyboard users track their navigation.

Maintain Sufficient Color Contrast

Ensure visual indicators (e.g., background fill, pseudo-checkbox) meet the minimum contrast ratio of 3:1.

Use accessible color combinations to support users with low vision or color blindness.

Best Practices

When to Use a Button

Use a button when the action:

Occurs on the same page (e.g., opening a modal, expanding content, submitting a form).

Triggers a UI change (e.g., toggling a tooltip, showing/hiding a section).

Performs an in-page interaction (e.g., closing a dialog, applying filters, playing audio).

Use a link (<a>) only when the action:

Navigates to a different page or URL.

Checkbox
Multi-select / Independent Boolean States
Common Issues

Focus indicator is missing or not clearly visible The checkboxes do not show a visible focus indicator when navigated via keyboard. This makes it difficult for keyboard users to know which checkbox is currently focused, especially for users with low vision or cognitive disabilities.

No accessible name due to missing label association Although labels are present visually, they are not programmatically associated with their corresponding checkboxes. This means screen readers may not announce the label when the checkbox receives focus.

Best Practices

Use Semantic Grouping for Related Checkboxes

Use <fieldset> and <legend> to group related checkboxes.

Alternatively, If above is not possible use role="group" with aria-label or aria-labelledby.

Ensure Explicit Label Association

Use for and id attributes to associate <label> with <input>.

This ensures screen readers announce the correct label.

Provide a Visible Focus Indicator

Ensure checkboxes show a clear visual change (e.g., outline or border) when focused.

This helps keyboard users track their navigation.

Associate Error Messages with Inputs

Use aria-describedby on the checkbox or group to link to the error message.

This ensures screen readers announce the error in context.

Chips
Tags / Filters / Input Tokens
Common Issues

Chips are not structured using list semantics The current implementation does not use <ul> and <li> or ARIA list roles (role="list" and role="listitem") to structure the chips. Without this, screen reader users may not understand that these are part of a group or collection.

Default chips lack role and keyboard support The default chips are not interactive, but if they are intended to be interactive (e.g., selectable or removable), they must have appropriate roles (like button or option) and support keyboard navigation (tabindex, Enter, Space).

Close buttons lack accessible name The close buttons are implemented using icons or spans without any accessible name or aria-label. Screen readers cannot announce what the button does (e.g., “Remove Lemon”).

Interactive chip text and close button are not both focusable If both the chip label and the close icon are interactive, they should both be independently focusable and have appropriate roles. Currently, only one may be reachable or neither clearly defined.

Best Practices

Use List Semantics for Chip Sets

Wrap chips in a <ul> with each chip as a <li>, or use role="list" and role="listitem" for ARIA-based implementations.

This helps screen readers understand the structure and count of items.

Ensure Keyboard Support for Interactive Chips

Use tabindex="0" for focusable chips.

Support Enter or Space for activation.

Use roving tabindex for arrow key navigation between chips.

Set all items except the first (or initially focused) to tabindex="-1", so only one element is focusable and tabbable.

Add keyboard event listeners to listen for arrow keys.

When user presses left/right (or up/down) arrows, move focus programmatically:

Remove tabindex="0" from the current element and set it to -1.

Set tabindex="0" on the new target element.

Modal & Dialog
Focus Trapping / Critical Interactions
Common Issues

Modal title is missing heading semantics The modal currently does not begin with a heading element. Every modal should start with a heading level 2 (<h2>) to provide a clear and accessible title for screen reader users. Without this, users may not understand the purpose or context of the modal content.

Best Practices

General Modal Guidelines

Use role="dialog" on the modal container

This helps assistive technologies recognize the element as a dialog.

Use aria-modal="true" for modal dialogs

This ensures screen readers treat background content as hidden and keep focus within the modal.

Label the modal using aria-labelledby or aria-label

Use aria-labelledby to reference the modal’s heading.

If no visible heading exists, use aria-label to provide a descriptive name.

Start modal content with a heading (<h2>)

This gives users a clear entry point and helps screen reader users understand the modal’s purpose.

Trap focus inside the modal

Ensure keyboard users cannot tab outside the modal while it is open.

Navigation Menu
Site Navigation / Submenus
Common Issues

Navigation and sub-navigation items are not structured using list semantics The main navigation sub-items are not wrapped in <ul> and <li> elements. Without this structure, screen readers cannot announce the number of items or their relationship

Link is nested inside a button In the “Last Sample” item, a link is placed inside a button. This causes the element to receive keyboard focus twice and creates confusion for screen readers and keyboard users. It also violates HTML semantics.

Avoid nesting links inside buttons: Use either a link or a button, not both. If an item needs to both navigate and expand, consider splitting the interaction or using a compound button pattern.

Best Practices

Best Practices for Accessible Navigation Menus

Use Semantic HTML:

Wrap navigation links inside a <nav> element with a meaningful aria-label.

Use list semantics with <ul> for the menu container and <li> for each navigation item.

Submenus must also use nested <ul> and <li> structures for clear hierarchy.

Expandable Menu Item Behavior:

Top-level items that expand submenus should be implemented with a separate button or control responsible for expanding/collapsing.

This separates the navigation link (which directs to a page) from the toggle that shows/hides the submenu.

The toggle must have aria-expanded to indicate open or closed state and an accessible name via aria-label like “Expand Sample submenu”.

Keyboard Interaction:

Only the navigation links and toggle buttons must be keyboard-focusable.

Arrow keys can be optionally (Not preferred) used for navigation within menus but tab order should be logical.

Pagination
Page Navigation Controls
Common Issues

Pagination items (1, 2, 3, etc.) are not structured using list semantics The numbered page links are not wrapped in <ul> and <li> elements. Without this structure, screen readers cannot understand the relationship between items or announce the total number of pages.

Next and Previous buttons do not reflect disabled state When the user is on the first or last page, the respective “Previous” or “Next” buttons are visually disabled but do not have the correct disabled attribute or aria-disabled="true" state. This can mislead screen reader users and keyboard users who may still try to interact with them.

“Go to page” accessible name does not match visual label The input field labeled “Go to page” has an accessible name that differs from its visible label. This inconsistency can confuse screen reader users.

“Go to page more” is not working The “Go to page more” image control is either non-functional or inaccessible, preventing users from navigating to additional pages beyond those currently visible.

Low color contrast on input field border The border color of the “Items per page” input field (#D1D3D4) on a background of #FAFAFA results in a contrast ratio of 1.439:1, which is far below the WCAG minimum requirement of 3:1 for graphical objects and UI components.

Best Practices

Structural Guidelines

Use list semantics for pagination items

Wrap page numbers (1, 2, 3, etc.) in a <ul> with each item in a <li>.

This helps screen readers understand the structure and count of pages.

Place combobox options adjacent to the control

Ensure dropdown options are placed immediately after the combobox in the DOM to maintain logical reading order.

Interaction & Keyboard Support

Use appropriate roles and states for navigation buttons (Next/Previous)

Use role="button" or <button> and aria-disabled="true" or “disabled” attribute when buttons are inactive.

Ensure buttons are keyboard-focusable when active and skipped when disabled.

Avoid redundant controls

If “Go” and “Submit” perform the same action, retain only one to reduce cognitive load.

Progress Bar
Dynamic Value Indicator
Common Issues

The progress bar is visually updating but lacks an accessible name or visible label to describe its purpose.

The implementation does not use semantic elements like <progress> or appropriate ARIA roles and attributes.

Best Practices

1. Use a Native or ARIA Role

Prefer using a <progress> element with min, max, and value attributes for best native support and compliance.

If a custom component is needed, assign role="progressbar" to the element.

2. Provide an Accessible Name

The progress bar must have a discernible accessible name.

Use aria-label for a concise description, e.g., aria-label="Download progress" or aria-labelledby referencing a visible label.

3. Communicate Value Changes

Use aria-valuenow to indicate the current value.

Use aria-valuemin and aria-valuemax for range boundaries.

When the value updates, the change must be announced to screen readers promptly.

4. Announce Dynamic Changes

To communicate new progress value dynamically, apply aria-live="polite" or aria-atomic="true" on a live region referenced in the label, or directly on the progress element if not using native <progress>.

Common Issues

The root container of the toolbar is missing role="toolbar" and lacks an accessible name.: Keyboard navigation is incorrect: Users must use Tab to move between buttons, instead of using arrow keys within the toolbar.

Subnavigation buttons:: Are missing expand/collapse state indicators (aria-expanded).

Missing toolbar role and name: Without role="toolbar" and an accessible name, screen readers cannot identify the region or its purpose.

Incorrect roles on buttons: Buttons are assigned role="listitem", which overrides native button semantics and confuses assistive technologies.

No grouping or toggle semantics: Buttons like Bold, Italic, Underline are not marked with aria-pressed, and mutually exclusive options are not grouped as radio buttons.

Best Practices

A toolbar is a container for grouping a set of controls. These controls are often related to tasks, such as formatting text in a text editor or navigating through application options. The toolbar role is used to identify such groups, helping users, particularly those using screen readers, to understand the structure and purpose of the controls within the toolbar.

It is often used in applications, editors, or dashboards and can include a combination of UI elements such as:

Radio groups (e.g., Text alignment: left, center, right)

1. Toolbar Container

Use role="toolbar" on the container element to identify it as a group of related controls.

Provide a clear and concise accessible name for the toolbar using aria-label or aria-labelledby.

If multiple toolbars exist on a page, each must have a unique and descriptive accessible name.

2. Keyboard Navigation

The entire toolbar container is focusable using Tab.

Inside the toolbar, users navigate between controls using arrow keys (Left, Right, Up, Down) depending on toolbar orientation (horizontal/vertical).

Only one child element within the toolbar has tabindex="0" (starting with the first or last focused element), all others have tabindex="-1".

Pressing arrow keys loops focus inside the toolbar and moves tabindex="0" accordingly.

Responsive Design
Reflow / Zoom / Orientation Adaptability
Best Practices

Content must be presented without two-dimensional scrolling (no horizontal and vertical scroll together) for:

Content designed to scroll vertically: It must fit within a 320 CSS pixel wide viewport.

Content designed to scroll horizontally: It must fit within a 256 CSS pixel high viewport.

Exceptions apply for truly two-dimensional content such as images, maps, diagrams, video, games, presentations, data tables, or interfaces requiring toolbars to remain in view.

Use CSS technologies like Flexbox, Grid, or other modern layout techniques to create responsive, flexible layouts that reflow content appropriately at different zoom levels and viewport sizes.

Text content on the page must resize up to 200% without loss of content or functionality (SC 1.4.4 Resize Text).

Use relative units like rem or em for font sizing rather than fixed units (px) to allow easy scaling.

The page must remain usable and readable at enlarged text sizes or zoom levels, without content overlap, clipping, or loss.

Browsers’ zoom features should be supported effectively, meaning users can zoom in without breaking layout or content visibility.

The page or application must respond seamlessly to both portrait and landscape orientations without locking orientation.

Content should adjust flexibly to different device orientations to maintain usability and readability.

After auditing all 24 components, five issues appeared repeatedly — across unrelated components, different interaction types, different years of implementation. Fix these across your entire design system, not just one component at a time.

#00B2A9 on white fails 3:1 contrast (2.64:1) Breadcrumbs, Accordion, Tabs, Datepicker Replace with a darker colour. This combination will never pass 3:1 for UI components on white.
No visible focus indicator on keyboard-focusable elements Accordion, Toggle Button, Checkbox, Chips, Toggle Switch, Tabs, Navigation Menu Add :focus-visible outline — minimum 2px, 3:1 contrast against adjacent colours. Test every interactive element by Tab key alone.

Master Checklist Universal Component Accessibility Checklist

Run this against every component before it ships. Use it in PR templates, audit reports, and QA sign-offs.

🏗️ Structure & Semantics

  • nav, main, header, footer landmarks present and each appears only once per page context
  • Navigation landmarks have unique aria-label values (Primary, Breadcrumb, Footer, Pagination)
  • Lists use <ul>/<ol>/<li> or role="list"/role="listitem" — never plain divs for grouped items
  • Only one H1 per page; heading hierarchy is unbroken (H1 → H2 → H3)
  • Tables use <th> with scope; never used for layout
  • Forms use <fieldset>/<legend> for groups; <label> associated via for/id on every input
  • Only one <header> and one <footer> per page

⌨️ Keyboard & Focus

  • All interactive elements reachable and fully operable by keyboard alone
  • Tab moves between components; Arrow keys navigate within components (tabs, toolbar, radio groups, chips)
  • Visible :focus-visible indicator on all elements — minimum 2px outline, 3:1 contrast
  • Modals trap focus while open; return focus to trigger on close
  • Roving tabindex used for tab panels, radio groups, toolbars, chips
  • No non-interactive element has tabindex="0"
  • Skip link present, visible on focus, placed before header, points to <main tabindex="-1">
  • Submenu links have tabindex="-1" when collapsed; tabindex="0" when submenu opens

🔖 ARIA Attributes

  • aria-expanded on all disclosure controls (accordions, nav toggles, comboboxes, dropdowns)
  • aria-controls links triggers to the ID of their controlled panel or listbox
  • aria-current="page" on active nav links and active pagination items
  • aria-hidden="true" on all decorative icons, separators, and visual-only elements
  • aria-describedby links error messages, hint text, and character limits to their inputs
  • aria-live regions exist for dynamic content (chip removal, result counts, progress, alerts)
  • aria-modal="true" on modal dialogs; aria-modal="false" or absent on non-modal
  • role="alert" containers start empty at page load; content injected dynamically to trigger announcement
  • aria-pressed only for standalone toggle buttons; aria-checked for radio and switch patterns

🎨 Colour Contrast

  • Text: ≥ 4.5:1 for small text; ≥ 3:1 for large text (18pt / 14pt bold+)
  • UI components: borders, icons, state indicators ≥ 3:1 against background
  • Focus indicator ≥ 3:1 against adjacent colours
  • Colour never the only means of conveying information — always paired with text, shape, or pattern
  • All interactive states tested: default, hover, focus, active, disabled, error, selected
  • #00B2A9 (2.64:1) not used on white for any interactive element

📝 Forms & Inputs

  • Permanent visible <label> on every input — not placeholder, not tooltip, not aria-label alone
  • Required fields: asterisk visible + aria-required="true" programmatic
  • Error messages visible, adjacent, linked via aria-describedby
  • Input borders ≥ 3:1 contrast against background
  • Read-only inputs have readonly attribute; disabled inputs have disabled attribute
  • Textarea character limits shown visually and announced via aria-live

📱 Responsive & Dynamic

  • No horizontal scrolling at 320px viewport width (WCAG 1.4.10 Reflow)
  • Text resizable to 200% without loss of content or function (WCAG 1.4.4)
  • Orientation not locked — content usable in portrait and landscape (WCAG 1.3.4)
  • Font sizes use rem or em — never fixed px
  • Links opening new windows include "opens in a new window" in accessible name
  • All interactive elements reachable and usable at 400% zoom
