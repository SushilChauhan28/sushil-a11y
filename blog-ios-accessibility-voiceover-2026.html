A few years ago, I was auditing a healthcare app for a mid-sized US client. The app let patients book doctor appointments, view prescriptions, and check lab results. Beautiful UI. Fast loading. Great reviews on the App Store.

Then I turned on VoiceOver.

The "Book Appointment" button announced itself as "button." Just that. Nothing else. The prescription list read out as a jumbled series of numbers with zero context. And the password field had no label at all. A blind user trying to manage their own healthcare had absolutely no idea what was happening on screen.

Accessibility is not a feature you add at the end. It is the foundation of a usable product.

01 — Introduction Why iOS Accessibility Cannot Be Optional

The Scale of the Problem

According to the World Health Organization, 1.3 billion people — roughly 15% of the global population — live with some form of disability. In the United States, the CDC estimates 61 million adults are affected. In India, the 2011 Census counted 2.68 crore persons with disability — widely considered an undercount.

These are not edge-case users. They are customers, patients, students, and employees who deserve equal access to digital services.

The Legal Landscape in 2026

United States: The ADA applies to mobile apps. Section 508 mandates accessibility for federal technology procurement — making VPAT documentation a hard requirement for government contracts.

European Union: The European Accessibility Act (EAA) came into force in June 2025. Mobile apps in banking, e-commerce, and transport must now meet EN 301 549 standards.

India: The RPWD Act 2016, Section 46, mandates accessible ICT. The legal framework is in place and increasingly cited in procurement requirements.

02 — VoiceOver Fundamentals — Gestures, Rotor, and the Reading Flow

💡

Try this before your next audit: Turn on VoiceOver, place your phone face-down, and try to complete a task using only what you hear. The discomfort you feel within thirty seconds is what your blind users experience in every session.

VoiceOver follows a linear, sequential navigation model. Users swipe right to advance, swipe left to go back, and double-tap to activate. There is no glancing at the whole screen — every piece of information must be discovered one element at a time.

The Reading Flow — What VoiceOver Announces

When VoiceOver focuses on an element, it reads information in a fixed sequence:

Label → Trait → Value → Hint

Well-implemented "Transfer Funds" button in a banking app:
"Transfer funds, button, available balance $3,450.00, double-tap to proceed"

Poorly implemented button:
"button" — no name, no context, no instruction.

03 — UIKit Accessibility Deep Dive

The Four Core Properties

Swift — UIKit
// Scenario: "Place Order" button in an e-commerce checkout
let placeOrderButton = UIButton()

// 1. Label — what VoiceOver announces as the element's name
placeOrderButton.accessibilityLabel = "Place order"

// 2. Hint — consequence of activation; full sentence with full stop
placeOrderButton.accessibilityHint = "Confirms and submits your order for delivery."

// 3. Value — current state or relevant context
placeOrderButton.accessibilityValue = "Total: £89.99, 2 items"

// 4. Traits — defines role and interactive behaviour
placeOrderButton.accessibilityTraits = [.button]

UIAccessibility Notifications

Swift — UIKit Notifications
// Scenario: Flight search form fails validation
// Error label appears — but VoiceOver focus stays on the Search button

// ❌ Broken — user never hears the error
self.dateErrorLabel.text = "Please select a departure date."
self.dateErrorLabel.isHidden = false

// ✅ Fixed — focus moves directly to the error
self.dateErrorLabel.text = "Please select a departure date."
self.dateErrorLabel.isHidden = false
UIAccessibility.post(
    notification: .layoutChanged,
    argument: self.dateErrorLabel
)

// Background success — announce without moving focus (WCAG 4.1.3)
UIAccessibility.post(
    notification: .announcement,
    argument: "Booking confirmed. Reference: FL-2094."
)

04 — SwiftUI Accessibility Best Practices

Images — Getting Them Right

Swift — SwiftUI Images
// Informative image — requires a descriptive label
Image("beachfront_villa_goa")
    .accessibilityLabel(Text("Beachfront villa in Goa with private pool and ocean view"))

// Decorative — must be hidden from VoiceOver
Image("card_shadow_effect")
    .accessibilityHidden(true)

// SwiftUI's built-in decorative initialiser
Image(decorative: "listing_background_gradient")

// SF Symbol as standalone action icon — always needs contextual label
Image(systemName: "location.fill")
    .accessibilityLabel(Text("View property on map"))
    .accessibilityAddTraits(.isButton)
    .onTapGesture { openMapView() }

// SF Symbol next to visible text — hide the icon entirely
HStack(spacing: 6) {
    Image(systemName: "bed.double.fill")
        .accessibilityHidden(true)   // "3 Bedrooms" text covers this meaning
    Text("3 Bedrooms")
}

Semantic Grouping — The Most Important Pattern

Without grouping, a hotel search result card with five pieces of information requires five separate swipes. Multiply that across twenty results and a blind user makes over a hundred swipes just to scan the list.

Swift — SwiftUI Semantic Grouping
// ❌ Without grouping — 5 separate elements, 5 swipes per card
VStack(alignment: .leading, spacing: 4) {
    Text("The Taj Mahal Palace")
    Text("5 stars")
    Text("0.3 km from city centre")
    Text("From ₹12,500 per night")
    Button("Book Now") { selectHotel(hotel) }
}

// ✅ With semantic grouping — 1 element, 1 swipe, full context
VStack(alignment: .leading, spacing: 4) {
    Text("The Taj Mahal Palace")
    Text("5 stars")
    Text("0.3 km from city centre")
    Text("From ₹12,500 per night")
    Button("Book Now") { selectHotel(hotel) }
}
.accessibilityElement(children: .ignore)
.accessibilityLabel(Text("The Taj Mahal Palace, 5 stars, 0.3 km from city centre, from 12,500 rupees per night"))
.accessibilityAddTraits(.isButton)
.accessibilityHint(Text("Double-tap to view hotel details and availability."))
.accessibilityAction { selectHotel(hotel) }

Dynamic Type — Non-Negotiable for WCAG 1.4.10

Swift — Dynamic Type
// ✅ Correct — system text styles scale with Dynamic Type automatically
Text("Account holder")
    .font(.subheadline)

// UIKit — these two lines are both required
referenceLabel.font = UIFont.preferredFont(forTextStyle: .body)
referenceLabel.adjustsFontForContentSizeCategory = true

// ❌ This fails WCAG 1.4.10 — fixed size never scales
referenceLabel.font = UIFont.systemFont(ofSize: 15)
⚠️

Test at maximum Dynamic Type size: Settings → Accessibility → Display & Text Size → Larger Text → drag to maximum (AX5). Your layout must remain readable and must not introduce horizontal scrolling.

Creating a VPAT/ACR Report for an iOS App

A VPAT (Voluntary Product Accessibility Template) is a standardised disclosure document required for US federal government procurement and increasingly requested by enterprise clients in healthcare, finance, and education. Use the VPAT 2.4REV template from itic.org.

For each WCAG criterion, assign one of four conformance levels:

  • Supports — fully meets the requirement with no known exceptions
  • Partially Supports — meets the requirement in some but not all cases; describe precisely what fails
  • Does Not Support — does not meet the requirement; describe the specific issue
  • Not Applicable — criterion does not apply to this product; justify clearly

The Remarks column is what separates a credible VPAT from a useless one:

Inadequate: "Partially Supports — some images may be missing labels."

Acceptable: "Partially Supports — product thumbnail images on category listing screens (screens 8–12) are missing descriptive accessibilityLabel values. VoiceOver announces the image file name. Remediation scheduled for version 4.1, targeted Q3 2026."

06 — Advanced Patterns and Common Pitfalls

Modal Focus Trapping

Swift — Modal Focus Trap
// UIKit — trap VoiceOver focus inside the modal
class PaymentConfirmViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()
        view.accessibilityViewIsModal = true
    }
}

// SwiftUI — applied to the modal's outermost container
VStack {
    Text("Confirm Payment")
    Text("Are you sure you want to pay £149.99?")
    HStack {
        Button("Cancel") { dismiss() }
        Button("Confirm") { processPayment() }
    }
}
.accessibilityAddTraits(.isModal)

Five Pitfalls I Encounter in Nearly Every Audit

  1. SF Symbol names read verbatim. Image(systemName: "square.and.arrow.up") announces as "square dot and dot arrow dot up." Always provide contextual labels.
  2. Icon and visible text both accessible. A share button with icon and "Share" text announces "square and arrow up, Share, button." Hide the icon with .accessibilityHidden(true).
  3. No heading structure on any screen. Users who navigate by headings find nothing in the Headings rotor. Add .accessibilityAddTraits(.header) to each screen's primary title.
  4. Form errors shown visually but never announced. Post a .layoutChanged notification directing focus to the error message immediately after it appears.
  5. Segmented control selected state not communicated. Apply the .selected trait dynamically to the active segment so VoiceOver announces "selected."

Complete iOS Accessibility Audit Checklist

VoiceOver Navigation

  • All interactive elements reachable by swiping in order
  • Focus order logical and matches reading sequence
  • Modals and overlays use .isModal / accessibilityViewIsModal
  • After modal dismissal, focus returns to a logical position
  • Each screen has at least one .header trait element

Labels and Descriptions

  • Every button and link has a meaningful accessibilityLabel
  • Decorative images hidden with .accessibilityHidden(true)
  • SF Symbol icons have explicit, contextual labels
  • Form fields have persistent programmatic labels
  • Icon + text combinations hide the icon to avoid duplication

State and Live Regions

  • Selected state communicated for tabs and segmented controls
  • Error messages announced via layoutChanged with focus moved
  • Success messages announced via .announcement notification
  • Loading states communicated to VoiceOver

Visual and Typography

  • All text meets 4.5:1 contrast (3:1 for large text)
  • App fully usable at maximum Dynamic Type size (AX5)
  • No horizontal scrolling at large text sizes
  • Focus indicator visible during Full Keyboard Access
